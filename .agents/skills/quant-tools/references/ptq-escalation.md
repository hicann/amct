# PTQ 升级

## 作用

这一页只讨论：

- 什么时候从直转量化升级到 PTQ
- 升级前做哪些粗粒度定位
- PTQ 应该如何从最小范围开始

## 什么时候进入 PTQ

默认满足下面条件后，再进入 PTQ：

1. 直转量化结果不满足要求
2. 已有 BF16 baseline
3. 已确认关闭量化后保持浮点等价
4. 已完成一轮粗粒度误差定位

## 粗粒度误差定位

这一阶段不追求精准根因，只追求缩小搜索空间。

优先回答：

- 是 MLP 更敏感，还是 attention 更敏感
- 是 dense 部分更敏感，还是 MoE 部分更敏感
- 是一开 activation quant 就明显掉点，还是 weight-only 已经不行
- 是整网普遍退化，还是集中在少数模块

这一步的产出应该是：

- 下一步优先尝试哪一类 PTQ 路线

## PTQ 引入原则

1. 先选最可能有收益的一类 PTQ 方法
2. 先在最小范围试，不一上来全开
3. 每轮只增加一档复杂度，避免同时改太多变量

## 最小 PTQ 实验

推荐先做：

- 一个 block
- 或一个 unit
- 或一种算法

并至少记录：

- 本轮相对上一轮的 PPL 变化
- 本轮新增了什么复杂度
- 当前是否缩小了问题范围

## 每轮 PTQ 后必须做的决策

每轮实验后，只允许做三种决策之一：

1. 停止
   - 当前结果已经够用
2. 补定位
   - 当前信息还不够，不继续叠复杂度
3. 小幅升级
   - 当前方向有效，再增加一档复杂度

## 不建议继续升级的情况

满足下面任一条时，不建议继续直接加算法：

- BF16 baseline 还不稳定
- 关闭量化后 wrapper 还不等价
- 当前实验没有提供新的定位信息
- 新增复杂度带来的收益不明显

## 长命令防超时

extract_ptq_data / ptq 在大模型上会超过 agent runtime 单条 Bash 超时上限（如 ~600s）被杀。务必**后台跑 + 轮询**：`nohup <命令> > run.log 2>&1 &` 启动后用多次短 Bash 调用轮询 run.log 到结束，不要阻塞式单跑。ptq 被中断用 `--start_block_idx <下一未完成层>` 续跑（逐层存参不重复），也可 `--start_block_idx/--end_block_idx` 主动分块，每块控制在超时窗口内。
